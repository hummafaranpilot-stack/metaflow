Phase 1: Image Compression & Resizing

Critical Impact

Images are 95%+ of page weight. Most are 700KB–2MB each when they should be 50–150KB. This single change can cut page weight by 80%+.

What to do:

1. Re-compress all WebP images with quality 70-80 (currently appear to be 90-100)
2. Resize images to their actual display dimensions:
  • Mobile hero: max 750px wide (currently serving desktop-size)
  • Product jars: max 400px wide
  • Thumbnails/icons: max 200px wide
  • KF Desktop hero-bg: max 1400px wide
3. Fix duplicate image requests on KW Mobile (remove trailing ? in src)
4. Use separate mobile-optimized images (smaller dimensions) for mobile pages
5. Consider AVIF format for browsers that support it (30-50% smaller than WebP)

Expected savings:
  KW Mobile: 15.3 MB → ~2.5 MB (save ~12.8 MB)
  KF Desktop: 21.2 MB → ~3.5 MB (save ~17.7 MB)
  KF Mobile: 19.0 MB → ~2.8 MB (save ~16.2 MB)

Pros

  • Biggest single improvement possible
  • No code changes needed — just replace image files

Cons

  • Slight visual quality reduction (usually unnoticeable at q75)
  • Need to re-process all images manually or via script
  • AVIF support not universal (need WebP fallback)
  • Must maintain separate mobile/desktop image sets

Phase 2: Deferred / Progressive Loading

High Impact

Even after compression, loading 80-150+ images at once is wasteful. Only load what the user can see, defer the rest.

What to do:

1. Add loading="lazy" to all images BELOW the fold
  • Keep hero image, first product shot, and trust badges eager
  • Lazy-load: testimonials, comparison charts, FAQ, footer, popups
2. Add fetchpriority="high" to the hero/LCP image
3. Preload the LCP image in <head> with <link rel="preload">
4. Defer popup & exit-intent images entirely (load via JS on trigger)
5. Use Intersection Observer for heavy sections (video embeds, galleries)
6. Defer third-party scripts (ClickBank badge, tracking pixels)

Expected LCP improvement:
  KW Mobile: 18.5s → ~4-6s (with Phase 1: ~2-3s)
  KF Desktop: 10.6s → ~3-5s (with Phase 1: ~1.5-2.5s)
  KF Mobile: 52.5s → ~8-12s (with Phase 1: ~2-3s)

Phase 3: Font & CSS Optimization

Medium Impact

9 font files (384 KB) and 29 KB unused CSS add render-blocking weight. Fonts block text rendering until downloaded.

What to do:

1. Reduce fonts from 9 to 2-3 (pick primary + accent)
  • Currently loading: Montserrat, Lato, Inter, Lora, Open Sans, Amariya, etc.
  • Consolidate to: Inter (body) + Montserrat (headings) or similar
2. Add font-display: swap to all @font-face rules
3. Preload critical fonts: <link rel="preload" as="font" crossorigin>
4. Convert any OTF fonts to WOFF2 (amariya-regular.otf = 98 KB, WOFF2 would be ~30 KB)
5. Subset fonts to only needed characters (Latin)
6. Remove unused CSS rules (29 KB identified by Lighthouse)
7. Inline critical CSS in <head>, defer the rest

Expected FCP improvement:
  KW Mobile: 3.2s → ~2.0s
  KF Mobile: 3.9s → ~2.2s

Pros

  • Faster text rendering — user reads content sooner
  • font-display: swap eliminates invisible text flash (FOIT)
  • WOFF2 compression is dramatically smaller than OTF

Phase 4: CLS & Layout Stability

Lower Impact (KF Desktop specific)

KF Desktop has CLS of 0.202 (target <0.1). Images without explicit dimensions cause layout shifts as they load.

What to do:

1. Add explicit width/height attributes to ALL <img> tags
  • KF Desktop: 47 images flagged as unsized
  • KW Mobile: also has unsized images
2. Use CSS aspect-ratio for responsive containers
3. Reserve space for dynamic content (pricing cards, badges)
4. Avoid injecting content above existing content after load
5. Set explicit dimensions on ad/third-party embeds

Expected CLS improvement:
  KF Desktop: 0.202 → <0.05
